Funciones de agregado

T08
¿Qué son?
Calculan un único valor a partir de varias filas. Se usan en el SELECT y a veces en el HAVING. No se pueden usar en el WHERE.
T08
COUNT
Cuenta filas. Hay tres variantes importantes:
COUNT(*) -- todas las filas, incluyendo NULLs COUNT(columna) -- solo filas donde columna no es NULL COUNT(DISTINCT c) -- valores únicos, sin repetir
Ejemplo real:
-- Cuántas provincias distintas tienen nuestros usuarios SELECT COUNT(DISTINCT provincia) FROM usuario;
Si hay 10 usuarios de Alicante → COUNT(DISTINCT provincia) cuenta Alicante solo 1 vez.
T08
SUM, AVG, MAX, MIN
SUM(cantidad) -- suma todos los valores AVG(pvp) -- promedio MAX(pantalla) -- valor máximo MIN(nacido) -- valor mínimo (con fechas: la más antigua)
Ejemplo real:
-- Total en euros de la cesta de un usuario SELECT SUM(pvp) total FROM cesta, articulo WHERE usuario='bmm@agwab.com' AND articulo=cod;
T08
ROUND
Redondea el resultado a N decimales. El segundo parámetro es el número de decimales.
SELECT ROUND(AVG(DISTINCT pvp), 2) mediaventa FROM articulo; -- ^--- 2 decimales
T08
Subconsulta para usar MAX/MIN en WHERE
No puedes poner MAX() directamente en el WHERE. Se resuelve con una subconsulta:
WHERE pvp = MAX(pvp) ← ERROR, esto no funciona
WHERE pvp = (SELECT MAX(pvp) FROM articulo) ← correcto
-- Artículo más caro SELECT cod, nombre, pvp FROM articulo WHERE pvp = (SELECT MAX(pvp) FROM articulo);

GROUP BY

T09
¿Para qué sirve?
Agrupa filas que tienen el mismo valor en una columna, para que los agregados (COUNT, SUM...) se calculen por separado para cada grupo.
Cuando el enunciado dice "por cada X" o "de cada X" → GROUP BY X
T09
Ejemplo básico
-- Cantidad de líneas por cada pedido SELECT numpedido, COUNT(*) FROM linped GROUP BY numpedido;
Cada fila de linped es una línea. GROUP BY numpedido junta las filas del mismo pedido y COUNT(*) las cuenta. Resultado: una fila por pedido.
T09
La regla de oro
Todo lo que aparece en el SELECT y NO es un agregado (COUNT, SUM...) DEBE estar en el GROUP BY.
SELECT marca, nombre, COUNT(*) FROM articulo GROUP BY marca; -- ERROR: 'nombre' no está en el GROUP BY
SELECT marca, nombre, COUNT(*) FROM articulo GROUP BY marca, nombre; -- Correcto
T09
GROUP BY con varias tablas
Cuando haces JOIN, el GROUP BY incluye todas las columnas del SELECT que no son agregados, aunque vengan de tablas distintas:
-- Cámaras pedidas: código, nombre, marca, total unidades y promedio importe SELECT l.articulo, a.nombre, a.marca, SUM(cantidad), AVG(importe) FROM linped l, articulo a WHERE l.articulo = a.cod AND articulo IN (SELECT cod FROM camara) GROUP BY l.articulo, a.nombre, a.marca;
T09
COUNT(*) vs COUNT(columna) con GROUP BY
-- Artículos pedidos incluyendo los que nunca se han pedido (salen con 0) SELECT cod, nombre, COUNT(numpedido) veces FROM articulo a LEFT JOIN linped l ON (a.cod = l.articulo) GROUP BY cod, nombre;
Aquí COUNT(numpedido) es clave: los artículos sin pedidos tienen numpedido = NULL en el LEFT JOIN, y COUNT(columna) no cuenta NULLs → salen con 0. Con COUNT(*) saldrían con 1.

HAVING

T10
¿Para qué sirve?
Filtra grupos después de agrupar. Es al GROUP BY lo que el WHERE es al SELECT.
WHERE filtra filas individuales. HAVING filtra grupos. WHERE va antes del GROUP BY, HAVING va después.
T10
Ejemplo claro
-- Marcas con más de 100 artículos, ordenadas por precio medio SELECT marca, AVG(pvp) promedio FROM articulo WHERE cod NOT IN (SELECT cod FROM pack) GROUP BY marca HAVING COUNT(*) > 100 ORDER BY promedio DESC;
WHERE elimina los packs antes de agrupar. HAVING filtra solo las marcas que, después de agrupar, tienen más de 100 artículos.
T10
HAVING con cálculos complejos
-- Provincias con promedio de importe total > 1500€ SELECT provincia, COUNT(DISTINCT p.numpedido) pedidos FROM usuario u, pedido p, linped l WHERE u.email=p.usuario AND p.numpedido=l.numpedido GROUP BY provincia HAVING AVG(importe*cantidad) >= 1500;
T10
Detectar duplicados con HAVING
Un uso muy común: encontrar valores repetidos en la tabla.
-- Apellidos que están repetidos SELECT apellidos FROM usuario GROUP BY apellidos HAVING COUNT(*) >= 2; -- Localidades que aparecen en 2 o más provincias SELECT pueblo, COUNT(*) provincias FROM localidad GROUP BY pueblo HAVING COUNT(*) >= 2;
T10
HAVING + WHERE + ORDER BY juntos
-- Meses con más de 10 pedidos: base imponible, IVA y total SELECT YEAR(fecha) año, MONTH(fecha) mes, ROUND(SUM(importe*cantidad)/1.21, 2) base, ROUND(SUM(importe*cantidad)/1.21*0.21, 2) IVA, ROUND(SUM(importe*cantidad), 2) total FROM pedido p JOIN linped l ON l.numpedido=p.numpedido GROUP BY YEAR(fecha), MONTH(fecha) HAVING COUNT(DISTINCT l.numpedido) > 10 ORDER BY YEAR(fecha), MONTH(fecha);

Orden de ejecución SQL

El orden en que SQL ejecuta cada cláusula
1. FROM → qué tablas (y los JOIN) 2. WHERE → filtra filas individuales 3. GROUP BY → agrupa las filas restantes 4. HAVING → filtra grupos 5. SELECT → calcula y muestra columnas 6. ORDER BY → ordena el resultado 7. LIMIT → recorta las filas mostradas
Por eso WHERE no puede usar COUNT o SUM: todavía no se han calculado. Eso va en HAVING.
Consecuencias prácticas
WHERE COUNT(*) > 2 ← ERROR: WHERE se ejecuta antes de GROUP BY
GROUP BY x HAVING COUNT(*) > 2 ← correcto
ORDER BY SUM(cantidad) ← puedes hacerlo, pero...
ORDER BY 2 ← "ordena por la 2ª columna del SELECT", más limpio
LIMIT — mostrar solo N filas
-- Los 5 usuarios que más han gastado SELECT u.email, u.nombre FROM usuario u JOIN pedido p ON u.email=p.usuario JOIN linped l ON l.numpedido=p.numpedido GROUP BY u.email, u.nombre ORDER BY SUM(importe*cantidad) DESC LIMIT 5;

Trucos y errores comunes

COUNT(*) vs COUNT(DISTINCT numpedido)
Cuando haces JOIN con linped y quieres contar pedidos, un pedido puede tener varias líneas → aparece varias veces. Usa COUNT(DISTINCT numpedido).
-- MAL: cuenta líneas, no pedidos SELECT fecha, COUNT(*) FROM pedido p, linped l WHERE p.numpedido=l.numpedido GROUP BY fecha; -- BIEN: cuenta pedidos distintos SELECT fecha, COUNT(DISTINCT l.numpedido) FROM pedido p, linped l WHERE p.numpedido=l.numpedido GROUP BY fecha;
Cálculo del IVA incluido
Cuando el precio ya lleva el IVA incluido (21%) y te piden separarlo:
-- importe*cantidad = precio total con IVA incluido base imponible = importe*cantidad / 1.21 IVA = importe*cantidad / 1.21 * 0.21 total = importe*cantidad
ROUND(SUM(importe*cantidad)/1.21, 2) -- base ROUND(SUM(importe*cantidad)/1.21*0.21, 2) -- IVA
GROUP BY forzado por ORDER BY
A veces necesitas GROUP BY no para mostrar un agregado, sino solo para ordenar por él:
-- Usuarios ordenados por lo que han gastado (sin mostrar el total) SELECT u.email, u.nombre FROM usuario u JOIN pedido p ON u.email=p.usuario JOIN linped l ON l.numpedido=p.numpedido GROUP BY u.email, u.nombre ORDER BY SUM(importe*cantidad) DESC;
IS NULL para valores vacíos
-- Artículos sin PVP SELECT COUNT(*) FROM articulo WHERE pvp IS NULL; -- Artículos sin PVP y descatalogados SELECT COUNT(*) FROM articulo a, stock s WHERE a.cod=s.articulo AND pvp IS NULL AND entrega='Descatalogado';
pvp = NULL no funciona. Siempre: pvp IS NULL
STD — desviación estándar
-- Media y desviación estándar de cantidad pedida por provincia SELECT p.codp, p.nombre, AVG(cantidad) media, STD(cantidad) desvest FROM provincia p, usuario u, pedido pd, linped l WHERE p.codp=u.provincia AND u.email=pd.usuario AND pd.numpedido=l.numpedido GROUP BY p.codp, p.nombre;

UNION y UNION ALL

PDF
¿Qué hace?
Junta los resultados de dos SELECT en uno solo. Las dos consultas deben tener el mismo número de columnas.
UNION elimina duplicados. UNION ALL los mantiene (y es más rápido).
T11
UNION — ejemplo básico
-- Artículos en cesta O en algún pedido SELECT articulo FROM cesta UNION SELECT articulo FROM linped;
Si un artículo está en los dos sitios, aparece solo una vez en el resultado.
T11
UNION ALL — cuando sabes que no hay duplicados
-- Artículos de pack y de cesta, aunque se repitan SELECT p.articulo, a.nombre FROM ptienea p JOIN articulo a ON p.articulo = a.cod UNION ALL SELECT c.articulo, a.nombre FROM cesta c JOIN articulo a ON c.articulo = a.cod;
Usa UNION ALL cuando los conjuntos son disjuntos (no puede haber duplicados). Es menos trabajo para la base de datos.
T11
UNION para clasificar usuarios por gasto
SELECT email, nombre, apellidos, 'GRAN CLIENTE' tipo FROM usuario JOIN pedido p ON p.usuario=email JOIN linped l ON l.numpedido=p.numpedido GROUP BY email, nombre, apellidos HAVING SUM(cantidad*importe) > 10000 UNION ALL SELECT email, nombre, apellidos, 'CLIENTE MEDIO' FROM usuario JOIN pedido p ON p.usuario=email JOIN linped l ON l.numpedido=p.numpedido GROUP BY email, nombre, apellidos HAVING SUM(cantidad*importe) BETWEEN 6000 AND 10000 UNION ALL SELECT email, nombre, apellidos, 'COMPRA POCO' FROM usuario JOIN pedido p ON p.usuario=email JOIN linped l ON l.numpedido=p.numpedido GROUP BY email, nombre, apellidos HAVING SUM(cantidad*importe) < 6000 ORDER BY 3;
Cada SELECT clasifica un grupo. Se unen con UNION ALL porque un usuario solo puede caer en una categoría → no hay duplicados.

INTERSECT

PDF
¿Qué hace?
Devuelve solo los valores que aparecen en AMBAS consultas. Es la intersección de conjuntos.
A INTERSECT B → solo lo que está en A y también en B.
T11
Artículos en stock, en cesta Y pedidos
SELECT articulo FROM stock INTERSECT SELECT articulo FROM cesta INTERSECT SELECT articulo FROM linped ORDER BY articulo;
Se pueden encadenar varios INTERSECT. Solo salen los artículos que están en los tres sitios a la vez.
T11
Pedidos que incluyen cámaras Y televisores
SELECT * FROM pedido WHERE numpedido IN ( SELECT numpedido FROM linped JOIN camara ON articulo = cod INTERSECT SELECT numpedido FROM linped JOIN tv ON articulo = cod );
Primero saca los pedidos con cámaras, luego los que tienen televisores. INTERSECT solo deja los que están en ambos.
T11
Artículos en cesta que el usuario ya compró
SELECT usuario, articulo FROM cesta INTERSECT SELECT p.usuario, l.articulo FROM pedido p JOIN linped l ON p.numPedido = l.numPedido;
Aquí INTERSECT trabaja con dos columnas a la vez: usuario y artículo deben coincidir en ambas consultas.

EXCEPT

PDF
¿Qué hace?
Devuelve lo que está en la primera consulta pero NO en la segunda. Es la diferencia de conjuntos.
A EXCEPT B → lo que está en A pero no en B. El orden importa.
T11
Artículos en cesta pero no en stock
SELECT articulo FROM cesta EXCEPT SELECT articulo FROM stock;
T11
Usuarios con pedidos pero sin dirección alternativa
SELECT usuario FROM pedido EXCEPT SELECT email FROM direnvio;
Saca todos los que han pedido y le quita los que tienen dirección alternativa. Lo que queda: los que pidieron pero nunca pusieron dirección alternativa.
T11
Encadenar EXCEPT
-- Artículos que no son pack ni han sido pedidos SELECT cod FROM articulo EXCEPT SELECT cod FROM pack EXCEPT SELECT articulo FROM linped;
Se aplican en orden: primero quita los packs, luego quita los pedidos. El resultado son artículos normales nunca pedidos.
T11
Usuarios que han pedido televisores pero nunca cámaras
SELECT email, nombre, apellidos FROM usuario u JOIN pedido p ON u.email=p.usuario JOIN linped l ON l.numpedido=p.numpedido JOIN tv ON tv.cod=l.articulo EXCEPT SELECT email, nombre, apellidos FROM usuario u JOIN pedido p ON u.email=p.usuario JOIN linped l ON l.numpedido=p.numpedido JOIN camara c ON c.cod=l.articulo;
Cada SELECT puede ser una consulta compleja con JOINs. EXCEPT resta el segundo resultado al primero.

EXISTS y NOT EXISTS

PDF
¿Qué hace?
No devuelve filas, solo comprueba si una subconsulta tiene resultados. Devuelve verdadero (1) o falso (0).
EXISTS → "existe alguna fila que cumpla esto". NOT EXISTS → "no existe ninguna fila que cumpla esto".
T11
Marcas con y sin artículos
-- Marcas QUE tienen artículos SELECT marca FROM marca m WHERE EXISTS (SELECT 1 FROM articulo a WHERE a.marca = m.marca); -- Marcas que NO tienen artículos SELECT marca FROM marca m WHERE NOT EXISTS (SELECT 1 FROM articulo a WHERE a.marca = m.marca);
El SELECT 1 dentro es irrelevante, solo importa si devuelve algo. La subconsulta usa m.marca de la consulta externa → es una subconsulta correlacionada.
T11
NOT EXISTS vs NOT IN — ojo con los NULLs
SELECT marca FROM marca WHERE marca NOT IN (SELECT marca FROM articulo); -- PELIGRO: si articulo.marca tiene NULLs, devuelve tabla vacía
SELECT marca FROM marca m WHERE NOT EXISTS (SELECT 1 FROM articulo a WHERE a.marca = m.marca); -- Correcto siempre, no le afectan los NULLs
Cuando hay NULLs en la columna de la subconsulta, NOT IN falla silenciosamente. NOT EXISTS siempre funciona bien. Usa NOT IN solo con listas de constantes o columnas que no admiten NULL.
T11
Artículo incluido en TODOS los pedidos
SELECT cod, nombre FROM articulo a WHERE NOT EXISTS ( SELECT 1 FROM pedido p WHERE NOT EXISTS ( SELECT 1 FROM linped l WHERE l.numpedido = p.numpedido AND l.articulo = a.cod ) );
Doble NOT EXISTS: "artículo tal que NO existe pedido que NO lo incluya". Es el patrón para la división relacional.
T11
EXISTS como comprobación
-- ¿Hay filas en la tabla marca? SELECT EXISTS (SELECT 1 FROM marca); -- Devuelve 1 (sí) o 0 (no) -- Mostrar 'sí' si hay filas SELECT 'sí' respuesta FROM dual WHERE EXISTS (SELECT 1 FROM marca);

NATURAL JOIN

PDF
¿Qué hace?
Une dos tablas automáticamente por todas las columnas que tienen el mismo nombre. No necesitas poner el ON.
Solo funciona bien cuando las columnas comunes son exactamente las que quieres usar para unir.
T11
Ejemplo básico
-- Equivalentes: SELECT * FROM articulo NATURAL JOIN memoria; SELECT * FROM articulo a, memoria m WHERE a.cod = m.cod;
Ambas hacen lo mismo porque la única columna común entre articulo y memoria es cod. Además, NATURAL JOIN no repite la columna común en el resultado.
T11
Cuándo falla — demasiadas columnas comunes
SELECT * FROM usuario NATURAL JOIN direnvio; -- Devuelve tabla VACÍA porque une por: email, calle, calle2, codpos, pueblo, provincia -- Es casi imposible que todos esos campos coincidan a la vez
usuario y direnvio comparten 6 columnas con el mismo nombre. NATURAL JOIN las usa todas → condición imposible → ninguna fila coincide.
T11
Cuándo produce producto cartesiano
SELECT * FROM cesta NATURAL JOIN pack; -- Produce producto cartesiano porque no hay columnas comunes -- cesta usa 'articulo', pack usa 'cod'
Si no hay columnas comunes, NATURAL JOIN se comporta como CROSS JOIN y genera todas las combinaciones posibles.

Trucos y errores comunes

Las columnas deben coincidir en número
SELECT cod, nombre FROM articulo UNION SELECT articulo FROM linped; -- ERROR: una tiene 2 columnas y la otra 1
SELECT cod FROM articulo UNION SELECT articulo FROM linped; -- ambas con 1 columna
ORDER BY solo al final de todo
SELECT articulo FROM cesta UNION SELECT articulo FROM linped ORDER BY articulo; -- solo aquí, al final del todo
No puedes poner ORDER BY en cada SELECT individual cuando usas UNION/INTERSECT/EXCEPT. Solo va una vez al final.
NOT IN peligroso con NULLs
-- Fechas de pedido que no coinciden con nacidos -- NOT IN falla porque nacido tiene NULLs: SELECT fecha FROM pedido WHERE fecha NOT IN (SELECT nacido FROM usuario); -- Devuelve 0 filas aunque debería devolver muchas -- Correcto con EXCEPT: SELECT fecha FROM pedido EXCEPT SELECT nacido FROM usuario ORDER BY fecha;
¿Cuándo usar cada operador?
UNIONesto O aquello, sin repetir
UNION ALLesto O aquello, con repetidos
INTERSECTesto Y aquello a la vez
EXCEPTesto pero NO aquello
EXISTS¿existe alguno que...?
NOT EXISTSseguro con NULLs, mejor que NOT IN

Apuntes tablas temporales T12 — Fundamentos de Bases de Datos UA

Subconsulta como columna

Teoría
¿Qué es?
Una subconsulta dentro del SELECT que devuelve un único valor (1 columna, 1 fila) y lo muestra como una columna más del resultado.
Siempre debe devolver exactamente 1 columna y 1 fila. Si devuelve más, da error.
T12
Mostrar varios conteos a la vez
Sin tablas temporales necesitarías varias consultas. Con subconsultas como columnas lo resuelves en una sola:
-- Cantidad de televisores, cámaras y objetivos en una sola fila SELECT (SELECT COUNT(*) FROM tv) tv, (SELECT COUNT(*) FROM camara) camaras, (SELECT COUNT(*) FROM objetivo) objetivos;
Cada subconsulta se ejecuta de forma independiente y devuelve un número. El resultado es una única fila con tres columnas.
T12
Subconsulta como columna calculada
Puedes usar el resultado para hacer cálculos con otros valores de la fila:
-- PVP de cada artículo y su % sobre el máximo global SELECT cod, pvp, pvp / (SELECT MAX(pvp) FROM articulo) * 100 AS pct FROM articulo WHERE pvp BETWEEN 1000 AND 1200;
La subconsulta calcula el MAX una vez y ese valor se usa en cada fila para calcular el porcentaje.
T12
Sustituir un valor por otro calculado
-- TVs Sony mostrando la pantalla media de TVs 1920x1080 en vez de la suya SELECT tv.cod, panel, (SELECT ROUND(AVG(pantalla),0) FROM tv WHERE resolucion='1920 x 1080') pantalla, resolucion, tdt FROM tv, articulo a WHERE tv.cod = a.cod AND marca = 'SONY';
La columna pantalla no muestra el valor real del televisor, sino el promedio calculado por la subconsulta. Útil para comparar cada fila contra un valor global.
T12
Porcentajes con subconsultas
-- % de TVs, cámaras y objetivos sobre el total de artículos SELECT (SELECT COUNT(*) FROM tv)/(SELECT COUNT(*) FROM articulo)*100 tvPC, (SELECT COUNT(*) FROM camara)/(SELECT COUNT(*) FROM articulo)*100 camarasPC, (SELECT COUNT(*) FROM objetivo)/(SELECT COUNT(*) FROM articulo)*100 objetivosPC;
Versión más eficiente: calcular el total una sola vez y reutilizarlo. Ver sección "Subconsulta en el FROM".

Subconsulta en el FROM

Teoría
¿Qué es?
Pones una consulta completa dentro del FROM y le das un alias. Se comporta como una tabla normal que puedes usar en el WHERE, JOIN, SELECT...
Siempre hay que darle un alias a la subconsulta del FROM, si no da error.
T12
Ejemplo básico — reutilizar un cálculo
-- Más eficiente: calcular total de artículos solo una vez SELECT (SELECT COUNT(*) FROM tv) / t.total * 100 tvPC, (SELECT COUNT(*) FROM camara) / t.total * 100 camarasPC, (SELECT COUNT(*) FROM objetivo) / t.total * 100 objetivosPC FROM (SELECT COUNT(*) total FROM articulo) t; -- ^ alias obligatorio
La subconsulta t devuelve una sola fila con el total. Luego se usa t.total en cada cálculo sin tener que repetir la consulta.
T12
Subconsulta en FROM como tabla intermedia
-- Provincias con promedio de importe de pedido > 2500€ SELECT provincia, COUNT(*) pedidos FROM usuario u, (SELECT usuario, p.numpedido, SUM(importe*cantidad) total FROM pedido p JOIN linped l ON p.numpedido=l.numpedido GROUP BY usuario, numpedido) pd WHERE u.email = pd.usuario GROUP BY provincia HAVING AVG(total) > 2500;
Primero la subconsulta pd calcula el importe total de cada pedido. Luego la consulta exterior agrupa por provincia y filtra con HAVING. Sin la subconsulta no podrías hacer AVG(total por pedido).
T12
Producto cartesiano entre subconsultas
-- Total alicantinos, los de Agost, y su porcentaje SELECT totalAlicante, deAgost, ROUND(deAgost/totalAlicante*100, 2) pct FROM (SELECT COUNT(*) totalAlicante FROM usuario WHERE provincia='03') t1, (SELECT COUNT(*) deAgost FROM usuario WHERE provincia='03' AND pueblo='0020') t2;
Cada subconsulta devuelve 1 fila × 1 columna. El producto cartesiano de dos tablas de 1 fila es... 1 fila. Es un truco habitual para combinar varios valores calculados.
T12
LEFT JOIN con subconsulta
-- Usuarios de provincia 03, y si tienen pedido > 10000€ mostrarlo SELECT email, nombre, apellidos, numpedido, valor FROM usuario u LEFT JOIN (SELECT p.numpedido, usuario, SUM(cantidad*importe) valor FROM pedido p, linped l WHERE p.numpedido = l.numpedido GROUP BY numpedido, usuario HAVING SUM(cantidad*importe) > 10000) calculo ON email = usuario WHERE provincia = '03' ORDER BY valor DESC;
El LEFT JOIN con la subconsulta hace que los usuarios sin pedidos grandes aparezcan igual, con NULL en numpedido y valor.

WITH — CTE (Common Table Expression)

Teoría
¿Qué es?
Es como una subconsulta en el FROM pero definida al principio, con nombre, y reutilizable varias veces en la misma consulta.
WITH nombreCTE AS ( SELECT ... -- la consulta temporal ) SELECT ... FROM nombreCTE ... -- la usas aquí
Ventaja: si necesitas usar el mismo cálculo en 3 sitios, lo defines una vez con WITH y lo reutilizas. Con subconsultas en el FROM tendrías que repetirlo.
T12
CTE básica — marca del artículo más caro
WITH mmax AS ( SELECT marca FROM articulo WHERE pvp = (SELECT MAX(pvp) FROM articulo) ) SELECT articulo, importe FROM linped l JOIN articulo a ON l.articulo = a.cod CROSS JOIN mmax WHERE a.marca = mmax.marca;
La CTE calcula la marca del artículo más caro. Luego la consulta principal la usa para filtrar. Sin CTE tendrías que repetir esa subconsulta.
T12
CTE para comparar con la media de su grupo
-- Artículos con PVP superior a la media de su propia marca WITH media_marca AS ( SELECT marca, AVG(pvp) AS media_pvp FROM articulo GROUP BY marca ) SELECT a.cod, a.marca, a.pvp, l.importe FROM articulo a JOIN media_marca mm ON a.marca = mm.marca JOIN linped l ON l.articulo = a.cod WHERE a.pvp > mm.media_pvp;
La CTE calcula la media por marca. Luego se une con articulo para comparar cada artículo contra la media de su propia marca. Esto sería muy difícil sin tabla temporal.
T12
Varias CTEs a la vez
-- Proporciones de TVs, cámaras y objetivos WITH t AS (SELECT COUNT(*) tvs FROM tv), c AS (SELECT COUNT(*) cams FROM camara), o AS (SELECT COUNT(*) objs FROM objetivo) SELECT tvs/(tvs+cams+objs) promtv, cams/(tvs+cams+objs) promcam, objs/(tvs+cams+objs) promobj, (tvs+cams+objs) total FROM t, c, o;
Se pueden definir varias CTEs separadas por comas después del WITH. Cada una tiene su nombre y todas están disponibles en la consulta final.
T12
CTE con cálculo en dos etapas
-- Por provincia: promedio de la máxima cantidad pedida en cada pedido WITH maximos AS ( SELECT numpedido, MAX(cantidad) cants FROM linped GROUP BY numpedido ) SELECT provincia, AVG(cants) FROM maximos m JOIN pedido p ON m.numpedido = p.numpedido JOIN usuario u ON p.usuario = u.email GROUP BY provincia;
Primero la CTE calcula el máximo de cantidad por pedido. Luego la consulta exterior calcula el promedio de esos máximos por provincia. Sin CTE no podrías hacer AVG sobre un MAX previamente calculado.
T12
La marca más vendida
-- Marca con mayor cantidad total vendida WITH ventas_marca AS ( SELECT a.marca, SUM(l.cantidad) AS unidades FROM articulo a JOIN linped l ON l.articulo = a.cod GROUP BY a.marca ) SELECT marca, unidades FROM ventas_marca WHERE unidades = (SELECT MAX(unidades) FROM ventas_marca);
La CTE se reutiliza dos veces: una en el FROM y otra en la subconsulta del WHERE. Sin CTE tendrías que escribir la misma consulta dos veces.

Pivotar una tabla

Teoría
¿Qué es pivotar?
Convertir filas en columnas. Por ejemplo, tener una fila por mes y convertirlo en una fila por usuario con una columna por cada mes.
-- Normal: una fila por mes mes | pedidos 10 | 5 11 | 3 12 | 8 -- Pivotado: una sola fila octubre | noviembre | diciembre 5 | 3 | 8
T12
Pivotar con subconsultas como columnas
-- Pedidos en oct, nov, dic de cualquier año SELECT (SELECT COUNT(*) FROM pedido WHERE MONTH(fecha)=10) octubre, (SELECT COUNT(*) FROM pedido WHERE MONTH(fecha)=11) noviembre, (SELECT COUNT(*) FROM pedido WHERE MONTH(fecha)=12) diciembre;
Cada columna es una subconsulta que filtra por un mes. Simple pero ineficiente si hay muchos meses porque recorre la tabla una vez por columna.
T12
Pivotar con CTE + IF — más eficiente
WITH contados AS ( SELECT usuario, MONTH(fecha) mes, COUNT(*) pedidos FROM pedido p JOIN usuario u ON p.usuario = u.email WHERE YEAR(fecha) = 2010 GROUP BY usuario, MONTH(fecha) ) SELECT usuario, MAX(IF(mes=1, pedidos, 0)) Ene, MAX(IF(mes=2, pedidos, 0)) Feb, MAX(IF(mes=3, pedidos, 0)) Mar, MAX(IF(mes=10, pedidos, 0)) Oct, MAX(IF(mes=11, pedidos, 0)) Nov, MAX(IF(mes=12, pedidos, 0)) Dic FROM contados GROUP BY usuario HAVING SUM(pedidos) > 1;
La CTE calcula los datos una vez. Luego IF(mes=1, pedidos, 0) devuelve el valor si es enero o 0 si no. MAX() se queda con el valor real (no el 0). Solo recorre la tabla una vez.
Teoría
IFNULL — quitar los NULLs del pivotado
-- Si un artículo no tiene pedidos en octubre, sale NULL -- IFNULL lo sustituye por vacío o por 0 IFNULL((SELECT veces FROM micte WHERE mes=10 AND micte.articulo=t2.articulo), 0) octubre
Cuando un artículo no aparece en un mes, la subconsulta no encuentra fila y devuelve NULL. IFNULL lo convierte en 0 o en lo que prefieras.

Trucos y patrones habituales

¿Cuándo usar cada técnica?
La regla general es: piensa el problema en etapas.
Si necesitas un único valor calculado → subconsulta como columna
Si necesitas una tabla intermedia que usar una vez → subconsulta en el FROM
Si necesitas reutilizar el mismo cálculo varias veces → WITH (CTE)
El patrón "dos etapas"
Muchos ejercicios de T12 siguen este patrón:
-- Etapa 1: calcular lo que necesitas como tabla temporal WITH etapa1 AS ( SELECT usuario, SUM(importe*cantidad) total FROM pedido p JOIN linped l ON p.numpedido=l.numpedido GROUP BY usuario ) -- Etapa 2: usar ese resultado para lo que realmente pides SELECT provincia, AVG(total) FROM usuario u JOIN etapa1 ON u.email = etapa1.usuario GROUP BY provincia HAVING AVG(total) > 2500;
Si intentas hacerlo todo en una consulta, AVG(SUM(...)) da error porque no puedes anidar agregados. La tabla temporal lo resuelve.
No se pueden anidar agregados
SELECT AVG(SUM(cantidad)) FROM linped GROUP BY articulo; -- ERROR: no puedes hacer AVG de un SUM directamente
-- Solución: tabla temporal WITH sumas AS ( SELECT articulo, SUM(cantidad) total FROM linped GROUP BY articulo ) SELECT AVG(total) FROM sumas;
Producto cartesiano de subconsultas de 1 fila
-- Útil para combinar varios valores calculados SELECT totalA, totalB, totalA/totalB*100 pct FROM (SELECT COUNT(*) totalA FROM tabla_a) a, (SELECT COUNT(*) totalB FROM tabla_b) b;
Cuando cada subconsulta devuelve exactamente 1 fila, el producto cartesiano también devuelve 1 fila. Es un patrón seguro y habitual en T12.
